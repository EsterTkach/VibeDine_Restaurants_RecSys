# VibeDine — Restaurant Recommendation System

## Project Overview

VibeDine is a personalized restaurant recommendation app. Every user gets a tailored feed of restaurants based on their taste profile, location, and activity history. The system uses a hybrid recommendation engine that blends content-based filtering (TF-IDF similarity) with collaborative filtering (item-based CF), with adaptive weighting based on how much the user has interacted with the app.

**Core features:**
- **Home carousels** — 5 dynamic recommendation rows on the main feed
- **Vibe Matcher** — modal where the user picks cuisine category + preferences (budget, distance, dietary, accessibility) to get a filtered recommendation
- **Group/Friends Matcher** — select friends from your list, set group preferences, get a restaurant that works for everyone

## Recommendation Engine

### Hybrid System (`api/services/recommendation_service.py`)

Combines content-based (CB) and collaborative filtering (CF) scores:

```
hybrid_score = alpha × CF_normalized + (1 - alpha) × CB_normalized
```

Alpha is determined dynamically per user based on interaction count:

| User likes | Alpha | Strategy |
|-----------|-------|---------|
| ≤ 5 | 0.0 | Pure content-based (cold start) |
| ≤ 20 | 0.4 | Mostly CB, some CF |
| ≤ 100 | 0.7 | CF-dominant |
| > 100 | 0.85 | Near-pure CF (warm user) |

If a user has zero likes, falls back to `get_popular_restaurants()` (weighted by avg_rating × review_count).

### Content-Based (`api/ml/cb_recommender.py`)
- TF-IDF vectorization on: `name + category + service_options + dining_options`
- `MAX_FEATURES = 10000`
- `compute_cb_scores(user_id)` → cosine similarity between liked items and all items, averaged across all liked items

### Collaborative Filtering (`api/ml/cf_recommender.py`)
- Weighted item-based CF: `score[i] = Σ(sim[liked_j, i] × rating[j]) / Σ(sim[liked_j, i])`
- Sparse CSR matrices (user-item + item-user)
- Online likes (in-app) augment the offline interaction matrix with rating=5

### Home Carousels (`api/routes/recommendations.py` → `GET /recommend/home-carousels`)

Returns 5 carousels:
1. **Recommended For You** — pure hybrid recs
2. **Popular Near You** — hybrid recs filtered by geospatial radius (user's lat/lon)
3. **Popular at this hour** — filtered by meal time (breakfast/lunch/dinner based on server time)
4. **You might like** — diversity carousel (items 25–50 from top-50 hybrid results)
5. **Hidden gems** — min_rating=4.5, max_reviews=30

### Group Recommendations (`api/services/groups_service.py`)

For a group of users:
1. Get top-50 hybrid recs per user independently
2. Aggregate: `avg_score = Σ(score) / count`
3. Coverage: `coverage = count / group_size` (% of group that received that restaurant)
4. `group_score = avg_score × coverage`
5. Return top-K by group_score

## Data Models

### Restaurant (MongoDB: `restaurants_collection`)
Key fields: `gmap_id`, `name`, `avg_rating`, `num_of_reviews`, `price`, `image_url`, `location` (GeoJSON Point), `hours`, `cuisines`, `establishment_types`, `meal_types`, `dining_styles`, `popular_items`, `dietary_preferences`, `is_accessible`

### User (MongoDB: `users_collection`)
Key fields: `user_id`, `username`, `password` (plaintext — see security note), `latitude`, `longitude`, `liked_restaurants`, `onboarding_preferences` (favorite_categories, favorite_atmospheres, accessibility, dietary_restrictions)

### User Interactions (MongoDB: `user_interactions_collection`)
Tracks: `liked_restaurants`, `saved_restaurants`, `viewed_restaurants`

### Offline Data
- `CF_interaction_matrix_2.parquet` — columns: `user_id`, `gmap_id`, `rating`; ratings ≥4 feed into CF training

## API Endpoints

### Recommendations
| Method | Endpoint | Description |
|--------|----------|-------------|
| GET | `/recommend/home-carousels?user_id=X&top_k=25` | 5 carousel rows for the home feed |
| GET | `/recommend/cf/{user_id}?top_k=10` | CF-only recommendations |
| GET | `/recommend/cb/{restaurant_name}?top_k=10` | Similar restaurants (content-based) |
| POST | `/recommend/group` | Group recommendation (body: `user_ids`, `filters`) |

### Users
| Method | Endpoint | Description |
|--------|----------|-------------|
| POST | `/users/signup` | Register |
| POST | `/users/login` | Login |
| POST | `/{user_id}/onboarding-preferences` | Save onboarding preferences |
| GET | `/{user_id}/onboarding-preferences` | Get preferences |
| POST | `/restaurants/{restaurant_id}/like` | Like a restaurant |
| DELETE | `/restaurants/{restaurant_id}/like` | Unlike |
| POST | `/{user_id}/restaurants/{restaurant_id}/save` | Save restaurant |
| POST | `/{user_id}/restaurants/{restaurant_id}/view` | Record view |
| GET | `/{user_id}/restaurants/liked` | Get liked restaurants |

## Notes for Contributors

- Demo users are hardcoded in `frontend/src/data/demoUsers.ts` — used as fallback when backend is unreachable
- Restaurant fallback data is in `frontend/src/data/restaurants.ts`
- All restaurant attributes (cuisines, atmosphere, etc.) are enumerated in `enums/models_enums.py` and mirrored as TypeScript enums in `frontend/src/types/index.ts`
- The `EXACT_CATEGORY_MAP` in `recommendation_service.py` maps user-facing category names (e.g. "sushi") to the raw Google Maps category strings stored in MongoDB
- Geospatial queries require a `2dsphere` index on `location` in the `restaurants` collection
